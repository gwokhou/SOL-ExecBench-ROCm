# Plan 172-01: Quant Readiness Triage

## Objective

Refine Quant readiness classification so the 33 Quant blockers distinguish true
CUDA/NVIDIA runtime dependencies from PyTorch ROCm-compatible semantic references
while preserving low-precision hardware-evidence boundaries. This plan modifies
only project-owned classifier logic in `inventory.py` and `readiness.py` -- it
does not mutate original or migrated dataset references.

## Background

The current readiness classifier has two overblocking paths for Quant problems:

1. **`_reference_runtime_hints` in `inventory.py`** performs a broad lexical scan
   of the reference text using `NVIDIA_RUNTIME_HINTS = ("cupy", "cuda.c",
   "cuda runtime", "nvrtc", "cublas", "cutlass")`. Any match populates
   `reference_runtime_hints` on the inventory record. This catches class names
   like `CuBLASRefBlockwiseGemm`, variable names like `scale_w_cublas`, and
   comments mentioning `cublas` -- all lexical false positives (D-03, D-07).

2. **`_low_precision_or_quant` in `readiness.py`** treats *all* Quant category
   problems as needing hardware evidence, regardless of whether the problem
   actually uses low-precision dtypes. Combined with `_blackwell_low_precision`
   routing the entire Quant category into the NVFP4/Blackwell path, many
   Quant problems that could run under PyTorch ROCm are stuck in
   `needs_hardware_evidence` (D-04, D-05, D-06).

## must_haves

### truths

- Original and migrated dataset reference names (e.g. `CuBLASRefBlockwiseGemm`,
  `scale_w_cublas`) are never renamed or mutated -- false positives are fixed
  in project-owned classifier logic only (D-07, D-08).
- Quant readiness improvement does not claim CDNA4 validation, low-precision
  hardware authority, or leaderboard parity (D-06, D-09, D-10, D-11).
- Static analysis is deterministic and does not execute dataset reference code
  during inventory classification.
- True CUDA-only Quant blockers record source evidence including problem id,
  file path, matched token, match kind, and context (D-12, D-13).

### deliverables

- Context-aware hint detection in `inventory.py` that distinguishes
  import/call/native-source evidence from lexical false positives.
- Reclassification in `readiness.py` that routes Quant problems into precise
  outcome classes: `ready`, `needs_hardware_evidence`, or
  `unsupported_nvidia_only_path`.
- Derived false-positive evidence records for Quant problems where lexical hints
  were ignored.
- CPU-safe tests covering all four Quant readiness scenarios (QUANT-01 through
  QUANT-04).

## Tasks

### Task 1: Add context-aware CUDA hint detection to inventory

<read_first>
- src/sol_execbench/core/dataset/inventory.py
- src/sol_execbench/core/dataset/readiness.py
- tests/sol_execbench/test_dataset_inventory_readiness.py
</read_first>

<action>
In `inventory.py`, replace the broad lexical `_reference_runtime_hints` function
with a context-aware hint classifier. The new logic must:

1. Define `NVIDIA_RUNTIME_BLOCKER_HINTS` as a tuple of hint tokens that remain
   true blockers: `"cupy"`, `"cuda.c"`, `"cuda runtime"`, `"nvrtc"`. These are
   runtime imports or native compilation paths that PyTorch ROCm cannot satisfy.

2. Define `NVIDIA_LEXICAL_FALSE_POSITIVE_HINTS` as a tuple of hint tokens that
   are *not* blockers when they appear only in comments, class names, variable
   names, docstrings, or compatibility labels: `"cublas"`, `"cutlass"`. These
   are semantic references that may be satisfied by PyTorch ROCm operations
   (e.g. `torch.matmul`, `torch.bmm`).

3. Create a new dataclass `ReferenceRuntimeHintEvidence` with fields:
   - `token: str` -- the matched hint token
   - `match_kind: str` -- one of `"import"`, `"call"`, `"native_source"`,
     `"solution_dependency"`, `"comment"`, `"class_name"`, `"variable_name"`,
     `"docstring"`, `"compatibility_label"`
   - `context: str` -- the surrounding line or short excerpt (max 200 chars)
   - `line_number: int | None` -- 1-based line number in reference text

4. Create a new function `_classify_reference_runtime_hints` that:
   - Scans reference text line by line
   - For `NVIDIA_RUNTIME_BLOCKER_HINTS`: matches any occurrence as a blocker
     (same behavior as current `_reference_runtime_hints`)
   - For `NVIDIA_LEXICAL_FALSE_POSITIVE_HINTS`: matches only when the token
     appears as part of a Python `import` statement, a function call (token
     followed by `(`), or a `.cu`/`.cuh` native source file reference
   - For matches that are only in comments (line starts with `#` or `"""`),
     class names (CamelCase identifiers containing the token), variable names
     (snake_case identifiers containing the token), or docstrings: record the
     evidence as a lexical false positive with the appropriate `match_kind`
   - Returns a tuple `(blocker_hints: list[str], false_positive_evidence:
     list[ReferenceRuntimeHintEvidence])`

5. Add `reference_runtime_false_positive_evidence: list[dict]` field to
   `ProblemDefinitionInventory` (default empty list). Each dict is the
   `dataclasses.asdict()` serialization of a `ReferenceRuntimeHintEvidence`.

6. Update `_definition_record` to call `_classify_reference_runtime_hints`
   instead of `_reference_runtime_hints`. Populate `reference_runtime_hints`
   with only `blocker_hints`, and populate the new evidence field with
   serialized false-positive evidence.

7. Keep `NVIDIA_RUNTIME_HINTS` as a backward-compatible alias (union of the two
   new tuples) so any other code that imports it continues to work.
</action>

- `ProblemDefinitionInventory` has `reference_runtime_false_positive_evidence`

### Task 2: Refine Quant readiness routing in readiness.py

<read_first>

- src/sol_execbench/core/dataset/inventory.py

<action>
In `readiness.py`, refine the Quant readiness routing to distinguish true
CUDA-only blockers from PyTorch ROCm-compatible references. Changes:

1. Add a new helper function `_quant_uses_low_precision_dtype` that checks
   whether a Quant workload actually uses low-precision dtypes (float8_e4m3fn,
   float8_e5m2, float4_e2m1, float4_e2m1fn_x2) in its input or output dtypes.
   Return `True` only when such a dtype is present.

2. Add a new helper function `_quant_uses_blackwell_format` that checks whether
   a Quant workload uses NVFP4/MXFP4/Blackwell-specific dtypes (float4_e2m1,
   float4_e2m1fn_x2, nvfp4, mxfp4) or the problem identity contains
   "blackwell", "nvfp4", or "mxfp4". Return `True` only when such evidence is
   present.

3. Modify the `classify_workload_readiness` function's NVIDIA blocker check
   (`elif _reference_has_nvidia_blocker(problem)`) to *not* block Quant
   category problems when the only `reference_runtime_hints` are lexical false
   positives (i.e. `reference_runtime_hints` is empty after Task 1's
   context-aware filtering). For Quant problems that still have true blocker
   hints, keep the existing `unsupported_nvidia_only_path` routing.

4. Modify the Quant/low-precision routing in `classify_workload_readiness`:
   - Replace the current `_blackwell_low_precision` check for Quant problems
     with a split:
     a. If `_quant_uses_blackwell_format` is True: route to
        `nvfp4_blackwell_specific` / `needs_hardware_evidence` with existing
        CDNA4 deferred evidence (same as current behavior for Blackwell).
     b. If `_quant_uses_low_precision_dtype` is True (but not Blackwell):
        route to `blocked_missing_evidence` / `needs_hardware_evidence` with
        the `low_precision_requires_hardware_evidence` code (same as current
        behavior for FP8).
     c. If neither: the Quant problem has standard dtypes (float16, float32,
        bfloat16, etc.) -- do NOT route to `needs_hardware_evidence`. Instead,
        let it fall through to the `ready` status with
        `pytorch_compatible` class, since the reference runs under PyTorch ROCm.

5. For Quant problems routed to `ready` (case 4c), add a reason with code
   `quant_rocm_compatible_reference` and message "Quant semantic reference is
   PyTorch ROCm-compatible; readiness does not imply low-precision hardware
   authority." This satisfies QUANT-02 and QUANT-04 by making the boundary
   explicit.

6. For Quant problems that have false-positive evidence (from Task 1) but are
   otherwise ready, add a reason with code `quant_cuda_false_positive_cleared`
   and message noting which lexical hint was ignored, referencing the evidence
   record.

7. Keep the existing `_low_precision_or_quant` function but remove the blanket
   `problem.category == "Quant"` check from it -- Quant category alone should
   not trigger low-precision routing. The function should now check only dtype
   presence.
</action>
