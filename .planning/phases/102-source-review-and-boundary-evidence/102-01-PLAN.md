# Phase 102 Plan 01: Source Review And Boundary Evidence

## Goal

Strengthen static source review coverage and evidence framing without expanding
the project claim from subprocess separation to hardened multi-tenant sandboxing.

## Scope

- Add an AST-aware Python review path in `reward_hack.py` for exploit families
  that raw regex scanning handles poorly.
- Expand regression tests for process, file, import, native loader, stream,
  cache, precision, string/comment false-positive, and obfuscation cases.
- Emit structured blocked-review payloads in source-review blocking messages.
- Update public and developer docs to state the actual execution boundary.

## Tasks

1. Add Python-aware static review helpers.
   - Parse `.py` source with `ast.parse`.
   - Detect risky imports, direct calls, attribute calls, dynamic import calls,
     `getattr(..., "system")` style process execution, loader calls, network
     calls, non-default stream calls, semantic cache patterns, and float32
     precision downgrades.
   - Keep regex scanning for native/non-Python sources and as fallback for
     syntactically invalid Python.
   - Preserve existing allow-list behavior for plain `os.environ`,
     `torch.compile`, HIP current-stream native text, and native C++ `data_ptr`.

2. Expand tests.
   - Assert Python strings and comments containing blocked words do not create
     false positives.
   - Assert additional obfuscated process/import/native loader/network/cache
     families are blocked.
   - Assert structured review evidence is available through `to_dict()` and the
     blocking message.

3. Update docs.
   - Add a visible README security-boundary note near usage.
   - Add a subprocess-boundary note in `docs/ARCHITECTURE.md`.
   - Keep wording precise: static review and subprocess execution are guardrails,
     not hardened sandboxing or multi-tenant isolation.

4. Verify and record completion.
   - Run focused reward-hack tests and eval-driver tests.
   - Run Ruff on touched Python files.
   - Add a phase summary and mark Phase 102 complete.

## Acceptance

- BOUNDARY-01: source-review tests cover broader bypass families.
- BOUNDARY-02: `.py` review uses AST-aware detection for Python constructs.
- BOUNDARY-03: blocked source-review outcomes include structured evidence in
  the blocking log message and review payload.
- BOUNDARY-04: README and architecture docs state the non-sandbox boundary.
