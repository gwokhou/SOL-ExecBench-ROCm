<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Parent profiling script → rocm-smi/amd-smi subprocess | Untrusted subprocess output crosses into trusted timing domain |
| timing_isolation.py → batch summary JSON | Environment data written to trusted evidence files |
| User-controlled environment → GPU state detection | Malicious environment could spoof detection results |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-176-01 | Spoofing | detect_concurrent_gpu_processes | mitigate | Validate subprocess output with regex parsing, timeout-bounded calls (5s), log unrecognized formats — prevents malicious process injection via SMI output spoofing |
| T-176-02 | Tampering | clock state verification | mitigate | Re-verify between problems (every 10) to detect drift, log warnings on state changes — prevents clock tampering during long batch runs |
| T-176-03 | Repudiation | batch summary environment snapshot | mitigate | Include schema_version, generated_at timestamp, and full tool probe results — provides non-repudiable audit trail |
| T-176-04 | Denial of Service | SMI tool hang | mitigate | 5-second timeout on all rocm-smi/amd-smi subprocess calls, graceful degradation to empty results — prevents SMI hangs from blocking profiling |
| T-176-05 | Information Disclosure | batch summary JSON | accept | Batch summary contains no sensitive credentials; GPU process list and clock state are benchmark metadata, not secrets |
| T-176-06 | Elevation of Privilege | sudo amd-smi calls (via clock_lock) | accept | Existing clock_lock.py already requires passwordless sudo; timing_isolation only reads state, does not elevate privileges |

**ASVS Level 1 Compliance:** All mitigations are standard input validation and timeout hygiene required by ASVS L1. No cryptographic controls needed (authentication/encryption out of scope).
</threat_model>
