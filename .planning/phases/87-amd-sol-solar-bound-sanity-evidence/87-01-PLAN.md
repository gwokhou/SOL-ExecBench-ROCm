<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local sidecar payloads -> report builder | Untrusted dictionaries from existing artifacts are summarized into strict diagnostic models. |
| source refs/checksums -> report output | User-controlled paths/refs/checksums are rendered into JSON and Markdown. |
| diagnostic report -> researcher interpretation | Human readers may mistake sanity evidence for validation or score authority unless boundaries are explicit. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-87-01-01 | Tampering | `build_amd_bound_sanity_report` | mitigate | Validate normalized output with strict Pydantic models, fixed status vocabulary, sorted serialization, bounded refs, and stable checksum. |
| T-87-01-02 | Repudiation | diagnostic status and claim boundaries | mitigate | Persist source statuses alongside sanity-specific statuses and explicit false authority/validation boundaries so reports show what was source evidence versus derived diagnostic risk. |
| T-87-01-03 | Information Disclosure | source refs and warnings | mitigate | Store compact path/ref/schema/checksum/warning summaries only; do not embed full traces, closures, Matrix reports, or score sidecars. |
| T-87-01-04 | Denial of Service | warnings and evidence gap examples | mitigate | Bound rendered example rows and sort/group warnings deterministically instead of copying unbounded payload arrays into Markdown. |
| T-87-01-05 | Elevation of Privilege | report interpretation | mitigate | Make score authority upgrade, leaderboard authority, model validation, native-host validation, and new-hardware validation literal false fields plus visible Markdown wording. |
| T-87-01-SC | Tampering | npm/pip/cargo installs | mitigate | No package-manager install tasks are planned; use existing Python/Pydantic/json/Markdown/checksum tooling only. |
