- `uv run scripts/report_consistency.py`, `uv run scripts/report_claim_upgrade.py`, and `uv run scripts/report_trust_summary.py` create diagnostic sidecar reports. Treat their outputs as diagnostic/review evidence only.

Canonical contracts that must not change:
- Trace JSONL remains the canonical run artifact.

<risks>
| Risk | Mitigation |
|------|------------|
| Release validation accidentally becomes a new benchmark authority surface | Keep wrapper as command composition and sidecar summary only; do not change canonical schemas or score semantics. |
| CPU-only host reports optional ROCm/Docker evidence as success | Use `unavailable`, `skipped`, or `deferred` statuses with explicit next actions for missing devices/tools. |
| Dataset run reuses stale traces | Use `--rerun` in release-candidate dataset-slice commands and require execution-closure artifact recording. |
| Logs expose local secrets or excessive environment details | Bound stdout/stderr tails and redact token-like values; never dump full environment maps. |
| Wording overclaims paper parity, leaderboard readiness, hard sandboxing, CDNA4, or MI300X/CDNA3 full-suite validation | Guard docs with tests and reuse phrasing from `docs/CLAIMS.md` and Phase 114 context. |
</risks>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| maintainer shell -> validation wrapper | CLI arguments select commands and output paths; validate bounded dataset limit and avoid shell interpolation. |
| validation wrapper -> subprocess commands | Existing pytest, Docker, dataset, and report scripts execute locally; capture bounded output only. |
| local environment -> shareable artifacts | Environment and logs can contain sensitive local context; redact token-like values and avoid full environment dumps. |
| optional ROCm/Docker evidence -> release claims | Hardware/container diagnostics must not upgrade canonical correctness, timing, score, paper-parity, leaderboard, or native-host authority. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-114-01 | Tampering | `scripts/release_candidate_validation.py` output paths | mitigate | Resolve output paths, create only under the requested output directory, and test deterministic artifact names. |
| T-114-02 | Information Disclosure | Summary JSON/Markdown log tails | mitigate | Cap stdout/stderr tails, redact token-like values, and omit full environment maps. |
| T-114-03 | Repudiation | Release validation result summary | mitigate | Record command argv, status, classification, artifact paths, and next action for every check. |
| T-114-04 | Spoofing | Optional Docker/ROCm evidence interpreted as release authority | mitigate | Encode evidence kind and authority boundaries in each optional result and docs. |
| T-114-05 | Denial of Service | Unbounded dataset validation | mitigate | Require explicit positive `--dataset-limit` and default to no dataset slice unless requested. |
| T-114-SC | Tampering | npm/pip/cargo installs | accept | No new package-manager installs are planned; this phase composes existing project dependencies. |
</threat_model>
