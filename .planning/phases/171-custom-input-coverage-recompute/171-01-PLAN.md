<objective>
Recompute RDNA4 readiness and coverage after Phase 170 custom input support, producing a complete before/after transition ledger for all 55 custom-input blockers and an updated coverage summary over the stable 235-problem denominator.

Purpose: Phase 170 made supported custom-input workloads ready to attempt. This plan measures the actual coverage movement by comparing the fixed baseline against a fresh readiness classification, then records every transition or residual blocker with precise classification. Without this, Phase 172-174 cannot know which readiness blockers were resolved and which remain.

Output: A transition ledger covering all 55 custom-input problems, an updated coverage report with revised blocker counts, and CPU-safe regression tests proving completeness and classification fidelity.
</objective>

    Create `scripts/build_custom_input_transition_ledger.py` that does the following:

    1. Load the fixed before-baseline from `out/rdna4-coverage-current/coverage.json` (per D-01). Extract the baseline checksum and record it in the output (per D-02). Record the baseline path as supplementary context (per D-03).

    2. Identify the cohort of 55 problems where `readiness_status == "custom_input_blocked"` in the baseline. These are the "original custom-input readiness blockers."

    3. Run a fresh readiness classification by calling `build_dataset_inventory` then `classify_rocm_readiness` against the current codebase (which now includes Phase 170 support). Then run `build_profiler_timing_coverage_report` to get updated coverage.

    4. For each of the 55 cohort problems, produce a problem-level transition record containing:
       - `problem_id`, `category`
       - `before_readiness_status`, `before_reason_codes` (from baseline)
       - `after_readiness_status`, `after_reason_codes` (from fresh classification)
       - `transition`: one of `promoted_to_ready`, `promoted_to_timing_fallback`, `promoted_to_profiler_backed`, `transitioned_to_runtime_blocked`, `transitioned_to_oom_blocked`, `transitioned_to_correctness_blocked`, `transitioned_to_profiler_blocked`, `residual_readiness_blocked`, or `no_change`
       - `workload_transitions_available`: boolean
       - `residual_class`: populated only when the after-disposition is still blocked, using one of the D-11 required residual classes (`unsupported_custom_entrypoint`, `gen_inputs_oom_blocked`, `gen_inputs_schema_mismatch`, `gen_inputs_device_mismatch`, `gen_inputs_timeout`, `execution_environment_unavailable`) or the existing `custom_input_requires_evaluator_support` for problems still missing a valid entrypoint.

    5. When workload-level evidence is available in the fresh `DatasetReadiness.workloads` for a cohort problem, populate a `workload_transitions` list with per-workload before/after status. When not available, set `workload_transitions_available: false` and include a `workload_transition_unavailable` marker per D-06.

    6. Write the output as `out/rdna4-custom-input-transition-ledger/transition-ledger.json` with a stable JSON checksum. Also write a human-readable `transition-summary.md`.

    7. Assert that `problem_denominator` in the fresh coverage report equals 235. Emit a warning and exit non-zero if it does not.

    The transition determination logic should map `before_readiness_status == "custom_input_blocked"` against the after status:
    - after `ready` -> `promoted_to_ready`
    - after `timing_fallback` -> `promoted_to_timing_fallback`
    - after `profiler_backed` -> `promoted_to_profiler_backed`
    - after `reference_oom_blocked` -> `transitioned_to_oom_blocked`
    - after `profiler_blocked` -> `transitioned_to_profiler_blocked`
    - after `readiness_blocked` with a different reason -> check specific reason codes for residual class
    - after `custom_input_blocked` with same reason -> `no_change`
    - any other blocked after status -> `residual_readiness_blocked`

    Also create `tests/sol_execbench/test_custom_input_transition_ledger.py` with CPU-safe tests:
    - Test that the 55-problem cohort is correctly extracted from a fixture baseline
    - Test that transition classification maps each before/after pair to the correct transition string
    - Test that `workload_transition_unavailable` is set when workload evidence is absent
    - Test that residual classes come from the D-11 required set or existing codes, never generic strings
    - Test that the 235-problem denominator assertion fails gracefully with a mismatched fixture
