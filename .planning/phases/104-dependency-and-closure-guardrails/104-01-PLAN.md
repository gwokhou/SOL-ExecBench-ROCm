# Phase 104 Plan 01: Dependency And Closure Guardrails

## Goal

Strengthen CPU-safe regression tests for dependency policy, execution closure
provenance, and hardware marker semantics.

## Scope

- Add dependency policy guardrails tying the default Docker target policy to
  project dependency declarations.
- Add execution closure tests for sidecar source refs, derived evidence cache
  provenance, checksum changes, and stable requested-evidence comparison.
- Add marker audit checks that timing, RDNA4, CDNA3, MI300X, and CDNA4 claims
  remain explicit and cannot be inferred from ordinary CPU-safe test success.

## Tasks

1. Extend dependency policy tests.
   - Assert the default selected Docker policy values are present in
     `pyproject.toml` and `uv.lock`.
   - Assert declared target policy IDs are unique and aligned with ROCm target
     local-version naming.

2. Extend execution closure provenance tests.
   - Assert `source_refs` can carry trace, score, timing, static evidence, and
     artifact-manifest refs.
   - Assert `provenance.derived_evidence` records sidecar/cache provenance and
     affects the closure checksum.
   - Assert requested evidence requirements compare order-insensitively to avoid
     stale-provenance false positives.

3. Extend marker audit tests.
   - Assert timing tests require explicit `-m timing_serial`.
   - Assert RDNA4 and CDNA3 are the only architecture-specific validation
     markers and that MI300X/CDNA4 remain claim/documentation boundaries, not
     marker shortcuts.

## Acceptance

- GUARD-01: dependency policy drift is guarded by manifest/project consistency
  tests.
- GUARD-02: closure provenance tests cover sidecar refs, stale-order handling,
  and manifest/cache provenance.
- GUARD-03: hardware marker tests make skipped hardware/timing behavior explicit
  and prevent MI300X/CDNA4 validation overclaims.
