<objective>
Add the core runtime evidence and compatibility-report layer for Phase 81.

Purpose: users need structured runtime evidence sidecars for declared ROCm Targets
without changing benchmark traces or score semantics. This plan creates a Python
helper that can collect or accept injected runtime evidence, write per-Target
Matrix Entry JSON, and aggregate entries into
`sol_execbench.rocm_compatibility_matrix.v1` reports with status counts.
</objective>

<task type="auto" tdd="true">
  <name>Task 1: Build scoped runtime evidence models and collectors</name>
  <files>src/sol_execbench/core/runtime_evidence.py, tests/sol_execbench/test_runtime_evidence_reports.py</files>
  <action>Add `runtime_evidence.py` with strict helper models for diagnostic runtime evidence categories and collection helpers. Reuse existing Matrix models from `compatibility.py` instead of defining a competing schema. Provide CPU-safe functions that construct `MatrixHostEvidence`, `MatrixContainerEvidence`, `MatrixPythonDependencyEvidence`, `MatrixToolchainEvidence`, `MatrixGpuEvidence`, and dependency policy evidence for a selected Docker Target. Host collection may inspect `/dev/kfd`, `/dev/dri`, Docker context/host values supplied by arguments, and nullable driver/ROCm strings. GPU collection should use PyTorch only when available and catch import/runtime failures; support injected values for tests. Record visible-device environment variables such as `HIP_VISIBLE_DEVICES`, `ROCR_VISIBLE_DEVICES`, `CUDA_VISIBLE_DEVICES`, and `GPU_DEVICE_ORDINAL`.</action>
  <acceptance_criteria>
    - Tests can construct scoped evidence without importing live ROCm-only packages.
    - Host, container, Python dependency, toolchain, dependency policy, and GPU evidence serialize in separate keys under `entry.model_dump(mode="json")["observed"]`.
    - Python evidence includes torch distribution/runtime version, local tag, HIP version, CUDA compatibility field, device availability, torchvision, and `triton-rocm` status.
    - GPU evidence includes device count, device name, gfx architecture, and visible-device environment variables when injected.
  </acceptance_criteria>
</task>

<task type="auto" tdd="true">
  <name>Task 2: Write per-Target sidecars and aggregate matrix reports</name>
  <files>src/sol_execbench/core/runtime_evidence.py, tests/sol_execbench/test_runtime_evidence_reports.py</files>
  <action>Add helpers to build a diagnostic `MatrixEntry` for a selected Target, write it as a per-target JSON sidecar, load one or more entry JSON files, and emit `RocmCompatibilityMatrixReport` JSON with validated `status_counts`. Add an argparse CLI with subcommands such as `collect-target` and `aggregate`. `collect-target` should accept `--manifest`, `--target`, `--output`, injected runtime evidence fields, dependency observation fields needed by Phase 80, optional failure-category fields, and `--allow-mixed-version-debug`. `aggregate` should accept repeated entry paths plus `--output`. All JSON writes should be deterministic with sorted keys and a trailing newline.</action>
  <acceptance_criteria>
    - Per-target JSON sidecar has schema version `sol_execbench.rocm_compatibility_matrix.v1`, target identity, observed evidence, status, reason code, reason, claim boundary, and artifacts.
    - Aggregate JSON has the same schema version, an ISO UTC `generated_at`, all entries, and exact counts by status.
    - CLI tests prove `collect-target` writes a sidecar and `aggregate` writes a report with expected status counts.
    - Invalid input exits nonzero without traceback where practical.
  </acceptance_criteria>
</task>

<task type="auto" tdd="true">

Write tests that create a canonical trace payload before and after sidecar generation and assert it is unchanged.</action>

    - Tests assert compatibility sidecar generation does not mutate a canonical Trace payload or require adding compatibility fields to Trace.

  </acceptance_criteria>
</task>
