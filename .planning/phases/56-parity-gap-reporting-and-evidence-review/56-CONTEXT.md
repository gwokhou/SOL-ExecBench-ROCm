# Phase 56: Parity Gap Reporting And Evidence Review - Context

**Gathered:** 2026-05-23
**Status:** Ready for research and planning
**Mode:** Autonomous defaults from roadmap, requirements, and completed Phase 53-55 artifacts

<domain>
## Phase Boundary

Phase 56 produces deterministic JSON and Markdown reports that explain dataset,
readiness, execution, scoring, and evidence gaps by category and suite.

The report consumes existing sidecar artifacts from prior phases:

- acquisition/layout manifest from Phase 53,
- inventory, readiness, and ready subset from Phase 54,
- execution closure from Phase 55,
- optional AMD-native score reports, AMD SOL v2 sidecars, SOLAR derivation
  sidecars, and ROCm timing evidence already generated by the runner.

This phase does not run GPU benchmarks, mutate canonical dataset files, mutate
trace JSON, create a hosted service, or claim full paper parity.
</domain>

<decisions>
## Implementation Decisions

### Locked

- Add a report-generation workflow, not a benchmark runner.
- Reports must be sidecar artifacts only: one deterministic JSON report and one
  deterministic Markdown report.
- Use the existing Python stdlib and local Pydantic/data helpers where useful;
  do not add dataframe, database, dashboard, or reporting framework
  dependencies.
- Complete denominators are required by category and suite: discovered,
  parsed, ready, blocked, not attempted, skipped, attempted, passed, failed,
  scored, degraded, and unscored.
- Blockers must be grouped by stable reason code and include concrete next
  actions from readiness where available.
- Evidence completeness must distinguish trace, timing, AMD-native score, AMD
  SOL, and SOLAR derivation evidence.
- Artifact references must be deterministic and path-safe. Paths should be
  relative to the report/output root when practical.
- Reports must preserve claim boundaries: not full 235-problem validation, not
  original 124-model extraction parity, not upstream SOLAR parity, not NVIDIA
  B200/Blackwell equivalence, not hosted leaderboard readiness, and not new
  CDNA3/CDNA4/NVFP4/MXFP4 validation.

### the agent's Discretion

- Exact module placement, helper names, and JSON field layout are at the
  implementer's discretion as long as tests pin the public report contract.
- The Markdown format may be concise and table-oriented; it should be readable
  in a terminal or PR without requiring a renderer.
- If optional artifacts are missing, report them as missing evidence rather than
  failing the whole report unless a required input sidecar is absent or malformed.
</decisions>

<specifics>
## Specific Ideas

- Add a small report module such as
  `src/sol_execbench/core/dataset/parity_gap.py` and a thin script such as
  `scripts/report_parity_gaps.py`.
- Inputs should include `--manifest`, `--inventory`, `--readiness`,
  `--ready-subset`, `--execution-closure`, optional `--amd-score-report`,
  optional derived sidecar directories, and output paths for JSON/Markdown.
- JSON should be stable under repeated runs with a fixed timestamp and should
  include source artifact checksums/provenance.
- Markdown should surface summary tables, blocker groups, evidence completeness,
  and claim boundaries without hiding failures or not-attempted items.
</specifics>
