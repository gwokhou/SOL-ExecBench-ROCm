## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Script startup → Lock acquisition | Process tries to acquire lock before any GPU work |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-175-01 | Tampering | Lock file path construction | mitigate | Use pathlib.Path for lock file path validation; call lock_file.parent.mkdir(parents=True, exist_ok=True) before open() |
| T-175-02 | Denial of Service | Parent directory permissions | accept | Lock file co-located with trusted output directory; if mkdir fails, OSError propagates with clear message |
| T-175-03 | Tampering | Stale lock after crash | accept | Kernel auto-release via fcntl.flock eliminates stale lock problem; no mitigation needed |
| T-175-04 | Information Disclosure | Lock file contains no sensitive data | accept | Lock file is empty (opened with "w" but never written to); no information disclosure risk |

**ASVS Level 1 Rationale:**
- V5 Input Validation: Path construction uses pathlib.Path; output directory exists check validates input
- No authentication, session management, or access control concerns (local trusted execution)
- Advisory locking is acceptable for trusted environments (same user runs scripts)
</threat_model>
