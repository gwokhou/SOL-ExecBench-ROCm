## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Report JSON -> diff loader | User-supplied Matrix report files are parsed and validated before comparison. |
| Diff renderer -> researcher interpretation | JSON/Markdown summaries may influence evidence claims and must not overstate authority. |
| CLI args -> filesystem | Script receives user-selected input and output paths. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-85-02-01 | Tampering | `load_matrix_report` | mitigate | Validate every input as `RocmCompatibilityMatrixReport`; reject duplicate diff keys and invalid authority fields instead of accepting malformed Matrix evidence. |
| T-85-02-02 | Repudiation | Diff output determinism | mitigate | Sort keys, severity categories, field groups, artifacts, and Markdown sections so repeated runs produce byte-identical outputs suitable for review. |
| T-85-02-03 | Information Disclosure | Markdown/JSON renderer | accept | Renderer emits only user-supplied Matrix report fields and bounded artifact refs; no hardware probes, credentials, Docker execution, or environment scans occur. |
| T-85-02-04 | Elevation of Privilege | Claim-boundary interpretation | mitigate | Always include diagnostic-only authority fields and Markdown wording that Docker/container evidence cannot become native-host validation, score authority, paper-parity authority, or leaderboard authority. |
| T-85-02-SC | Tampering | npm/pip/cargo installs | mitigate | No package-manager install tasks are planned; use existing Python/Pydantic/json/Markdown tooling only. |
</threat_model>
