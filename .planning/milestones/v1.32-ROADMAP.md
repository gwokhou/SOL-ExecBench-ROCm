## Requirements

### RDNA4-PROF-COV-01: 235-Problem Timing Coverage Accounting

The project must be able to generate a problem-level coverage report over the
full expected 235-problem denominator. Each problem must be classified as one
of:

- `profiler_backed`
- `timing_fallback`
- `ready_missing_profiler_timing`
- `readiness_blocked`

### RDNA4-PROF-COV-05: Replacement Classification Statuses

The profiler timing coverage report must distinguish partial and blocked
profiler replacement attempts from plain fallback timing:

- `partial_profiler_backed` means `rocprofv3` kernel activity rows were
  collected, but the replacement does not cover every workload successfully.
- `profiler_blocked` means a profiler replacement attempt exists but did not
  produce usable `rocprofv3` kernel activity evidence.

Neither status may count toward `profiler_backed` coverage or full timing
coverage.

### RDNA4-PROF-COV-06: Classified Resume Semantics

The replacement runner must avoid repeatedly profiling already classified
full-problem replacement attempts while still retrying workload-limited smoke
artifacts, because smoke artifacts are not full-problem classifications.

### RDNA4-PROF-COV-07: OOM-Safe Continuation Controls

The replacement runner must support explicit skip lists and configurable
staging temp roots so large RDNA4 profiler continuations can avoid known OOM
targets and avoid filling the root `/tmp` filesystem.

### RDNA4-PROF-COV-08: Blocked Profiler Classification Sidecars

Known OOM, timeout, and environment-blocked profiler attempts must be
classifiable as `profiler_blocked` sidecars so the 235-problem denominator
remains auditable without repeatedly re-running unsafe targets.

### RDNA4-PROF-COV-09: Workload-Sharded Profiler Triage

The replacement runner must support workload offset controls so problematic
RDNA4 profiler targets can be triaged in small workload slices.

### RDNA4-PROF-COV-10: Slice Claim Boundary

Workload-sliced profiler sidecars must record slice metadata and must not be
treated as full-problem classifications by resume or full profiler-backed
coverage accounting.

### RDNA4-PROF-COV-11: Workload-Sharded Aggregation Contract

The profiler timing pipeline must be able to execute each expected workload of
a problem as an independent `rocprofv3` unit, record deterministic workload ids
and source trace paths, and aggregate those units into one problem-level
profiler-backed timing sidecar only when every expected workload has passed and
has usable profiler-backed kernel timing.

### RDNA4-PROF-COV-12: OOM/Timeout Structural Mitigation

OOM and timeout mitigation must come primarily from bounded workload-level
execution, resumable manifests, and retryable failure classification. Process
memory limits and long timeouts may remain as circuit breakers, but they must
not be the only mechanism required to make large profiler timing problems
complete.

### RDNA4-PROF-COV-13: Remaining Sharded Closure Audit

The project must provide a deterministic audit for remaining
`partial_profiler_backed` and `profiler_blocked` RDNA4 timing targets that
records problem id, workload count, failure reason, existing manifest state,
and a recommended next action without counting those targets as profiler-backed
coverage.

### RDNA4-PROF-COV-14: Conservative Sharded Closure Targeting

The generalized sharded closure workflow must default to partial/blocked
profiler targets only. It must not schedule readiness-blocked or broad fallback
targets unless explicitly requested, and it must preserve claim boundaries for
partial and imported diagnostic evidence.

### RDNA4-PROF-COV-15: Partial Failure Mode Classification

The project must classify `partial_profiler_backed` targets by observed
workload failure mode, including invalid reference, runtime error, profiler
evidence missing, mixed correctness failure, and all-runtime-error patterns.

### RDNA4-PROF-COV-16: Profiler Closure Decision Ledger

Each partial profiler target must have a deterministic closure decision that
states whether it is profiler-closable, blocked on correctness/runtime
behavior, blocked on profiler evidence, or requires manual denominator
reclassification.

### RDNA4-PROF-COV-17: Reference OOM Denominator Status

The profiler timing coverage report must distinguish current-device
reference/gen_inputs HIP OOM failures from profiler lifecycle failures. A
problem with profiler replacement evidence whose non-PASSED trace logs show
reference or input-generation OOM on the current RDNA4 device must be
classified as `reference_oom_blocked`.

### RDNA4-PROF-COV-18: Reference OOM Claim Boundary

`reference_oom_blocked` problems must remain inside the 235-problem denominator
and must not count as `profiler_backed` or full profiler-backed timing
coverage. Reports must preserve the distinction between denominator
accounting and complete profiler-backed timing coverage.

### RDNA4-PROF-COV-02: Profiler-Backed Claim Boundary

Only timing sidecars with `profiler_collected=true` and `rocprofv3` backend may
count as profiler-backed timing coverage. Device-event, PyTorch profiler, or
other fallback timing sidecars must remain visible but must not satisfy full
profiler-backed coverage.

### RDNA4-PROF-COV-03: Fallback Replacement Runner

The project must provide a runner that targets current `timing_fallback`
problems from the coverage ledger, profiles their staged `eval_driver.py`
processes directly under `rocprofv3`, and writes replacement sidecars under a
problem-level timing evidence root compatible with the coverage report.

### RDNA4-PROF-COV-04: Incremental Long-Run Controls

The replacement runner must support bounded and resumable execution so the 121
problem batch can be run in chunks without losing already collected
profiler-backed timing sidecars.

## Active Guardrails

- RDNA4 validation jobs may run for many hours. Poll, checkpoint, and preserve
  logs/artifacts; do not terminate healthy long-running processes solely due to
  duration.
- Long RDNA4 timing or derived jobs that can exhaust memory must run through
  isolated `systemd-run --user` units or equivalent memory/swap caps.
- This milestone may improve timing coverage accounting and batch readiness,
  but public RDNA4 validation claims must not be upgraded until full evidence
  is archived.
