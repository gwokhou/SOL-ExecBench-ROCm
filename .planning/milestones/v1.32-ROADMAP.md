## Requirements

### RDNA4-PROF-COV-01: 235-Problem Timing Coverage Accounting

The project must be able to generate a problem-level coverage report over the
full expected 235-problem denominator. Each problem must be classified as one
of:

- `profiler_backed`
- `timing_fallback`
- `ready_missing_profiler_timing`
- `readiness_blocked`

### RDNA4-PROF-COV-05: Replacement Classification Statuses

The profiler timing coverage report must distinguish partial and blocked
profiler replacement attempts from plain fallback timing:

- `partial_profiler_backed` means `rocprofv3` kernel activity rows were
  collected, but the replacement does not cover every workload successfully.
- `profiler_blocked` means a profiler replacement attempt exists but did not
  produce usable `rocprofv3` kernel activity evidence.

Neither status may count toward `profiler_backed` coverage or full timing
coverage.

### RDNA4-PROF-COV-06: Classified Resume Semantics

The replacement runner must avoid repeatedly profiling already classified
full-problem replacement attempts while still retrying workload-limited smoke
artifacts, because smoke artifacts are not full-problem classifications.

### RDNA4-PROF-COV-02: Profiler-Backed Claim Boundary

Only timing sidecars with `profiler_collected=true` and `rocprofv3` backend may
count as profiler-backed timing coverage. Device-event, PyTorch profiler, or
other fallback timing sidecars must remain visible but must not satisfy full
profiler-backed coverage.

### RDNA4-PROF-COV-03: Fallback Replacement Runner

The project must provide a runner that targets current `timing_fallback`
problems from the coverage ledger, profiles their staged `eval_driver.py`
processes directly under `rocprofv3`, and writes replacement sidecars under a
problem-level timing evidence root compatible with the coverage report.

### RDNA4-PROF-COV-04: Incremental Long-Run Controls

The replacement runner must support bounded and resumable execution so the 121
problem batch can be run in chunks without losing already collected
profiler-backed timing sidecars.

## Active Guardrails

- RDNA4 validation jobs may run for many hours. Poll, checkpoint, and preserve
  logs/artifacts; do not terminate healthy long-running processes solely due to
  duration.
- Long RDNA4 timing or derived jobs that can exhaust memory must run through
  isolated `systemd-run --user` units or equivalent memory/swap caps.
- This milestone may improve timing coverage accounting and batch readiness,
  but public RDNA4 validation claims must not be upgraded until full evidence
  is archived.
