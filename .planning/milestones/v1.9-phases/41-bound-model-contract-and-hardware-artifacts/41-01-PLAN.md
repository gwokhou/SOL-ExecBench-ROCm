<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| external JSON path -> loader | Untrusted local JSON enters strict parser through `load_amd_hardware_model(path)` |
| packaged JSON -> defaults | Packaged resource JSON becomes trusted default model evidence only after validation |
| architecture token -> package resource | User/API-provided architecture selects a packaged resource name |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-01 | Tampering | `amd_hardware_model_from_dict` | mitigate | Reject unknown fields, old `validation_status`, missing provenance, non-positive values, unknown enums, and architecture/status policy violations with `ValueError`. |
| T-41-02 | Repudiation | `gfx1200.json` provenance fields | mitigate | Require source/evidence fields and split `hardware_validation_status` / `model_validation_status` so claims are auditable. |
| T-41-03 | Tampering | `load_packaged_amd_hardware_model` | mitigate | Use architecture tokens with expected architecture checks; do not accept arbitrary resource path fragments as packaged resource names. |
| T-41-04 | Denial of Service | external JSON parsing | accept | Local file loading is maintainer-operated; parser reads one small JSON artifact and fails explicitly on invalid content. |
| T-41-SC | Tampering | package installs | accept | No npm/pip/cargo package installs are planned; no package legitimacy checkpoint is required. |
</threat_model>
