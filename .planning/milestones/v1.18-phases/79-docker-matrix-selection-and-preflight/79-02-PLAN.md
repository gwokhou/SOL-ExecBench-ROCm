<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI flags to shell parser | User-provided Target and override flags control Docker image selection. |
| Python helper output to shell build/run | Structured selection/preflight JSON becomes shell variables and build args. |
| host device/context state to execution gate | Docker context and ROCm device state determine whether build/run may proceed. |
| Docker wrapper to host resources | The container receives selected host devices and project mount. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-79-06 | Tampering | `--target` parsing | mitigate | Resolve Target ids only through the Plan 79-01 manifest helper; reject unknown ids without explicit override. |
| T-79-07 | Elevation of privilege | `docker run` flags | mitigate | Preserve targeted `/dev/kfd` and `/dev/dri` device mounts and existing group/security options; do not add `--privileged` or new broad mounts. |
| T-79-08 | Spoofing | diagnostic output | mitigate | Emit bounded Phase 78 statuses/reason codes and authority-false fields; never label script selection or preflight as clean validation. |
| T-79-09 | Denial of service | unavailable Docker/ROCm runtime | mitigate | Classify as `runtime_unavailable` before build/run/benchmark execution and exit deterministically. |
| T-79-10 | Information disclosure | diagnostic output | accept | Diagnostics contain Docker context/device status and requested image/build args only; no credentials or proprietary kernels are emitted. |
| T-79-SC | Tampering | package installs | accept | No new package installs; script uses existing Bash, Python module, Docker CLI, and project dependencies. |
</threat_model>
