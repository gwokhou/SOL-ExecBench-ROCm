## Phase Details

### Phase 73: Static Evidence Contract And Guardrails
**Goal**: Maintainers and consumers have a strict diagnostic-only static evidence contract before artifact collection or reporting exists.
**Depends on**: Phase 72
**Requirements**: SKE-CONTRACT-01, SKE-CONTRACT-02, SKE-CONTRACT-03, SKE-CONTRACT-04, SKE-CONTRACT-05
**Success Criteria** (what must be TRUE):
  1. Maintainer can serialize and parse a strict `sol_execbench.static_kernel_evidence.v1` sidecar with stable status, reason-code, artifact, tool-run, and classification fields.
  2. Consumer can inspect authority metadata showing static evidence is diagnostic only and is not correctness, performance, timing, score, paper-parity, or leaderboard authority.
  3. Maintainer can represent aggregate and per-artifact collected, partial, unavailable, unsupported, failed, and skipped states.
  4. Consumer can discover optional static evidence support from evaluator contract metadata without a required evaluator contract version bump.
  5. Maintainer can verify static evidence helpers leave canonical trace JSONL, correctness, timing, scoring, and default benchmark behavior unchanged.
**Plans**: TBD

### Phase 74: Build Artifact Discovery And Manifest
**Goal**: Operators can archive static artifacts from the exact current HIP/C++ solution build before temporary staging cleanup.
**Depends on**: Phase 73
**Requirements**: SKE-ARTIFACT-01, SKE-ARTIFACT-02, SKE-ARTIFACT-03, SKE-ARTIFACT-04, SKE-ARTIFACT-05
**Success Criteria** (what must be TRUE):
  1. Operator can collect static evidence only from the current HIP/C++ staging or build tree, starting with `benchmark_kernel.so` and opportunistic code objects, HSACO files, object files, and compiler outputs.
  2. Operator can find persisted copies of discovered static artifacts under an output-derived evidence directory after staging cleanup.
  3. Operator can inspect an artifact manifest containing artifact kind, source path, persisted path, SHA256, size, producer, target architecture when known, and inspectability for every registered artifact.
  4. Maintainer can verify discovery never scans global caches, ROCm install trees, or unrelated temporary directories that could mix artifacts across benchmark runs.
  5. Operator receives explicit unsupported or unavailable states for solution paths without a stable v1.17 static artifact boundary.
**Plans**: TBD

### Phase 75: Routed Static Extractor Adapters
**Goal**: Operators can extract static ISA and metadata through routed, bounded, nonfatal static-analysis tools.
**Depends on**: Phase 74
**Requirements**: SKE-EXTRACT-01, SKE-EXTRACT-02, SKE-EXTRACT-03, SKE-EXTRACT-04, SKE-EXTRACT-05, SKE-EXTRACT-06
**Success Criteria** (what must be TRUE):
  1. Operator can see static extraction routed through the v1.16 toolchain layer instead of direct ad hoc executable lookup.
  2. Operator can receive bounded `llvm-objdump` output when routing selects an available compatible tool.
  3. Operator can receive bounded `readelf` metadata output as a fallback or complementary route when available.
  4. Operator can inspect route decisions, selected and unavailable tools, command provenance, timeout, return code, stdout/stderr tails, and raw output artifact paths for every attempted extractor.
  5. Operator receives nonfatal failed, partial, or unavailable sidecars when tools are missing, unsupported, time out, return nonzero, or produce unparseable output.
**Plans**: TBD

### Phase 76: CLI Sidecar Integration And Reports
**Goal**: Operators can opt into static evidence during benchmark evaluation and inspect the resulting sidecars and summaries without changing benchmark semantics.
**Depends on**: Phase 75
**Requirements**: SKE-CLI-01, SKE-CLI-02, SKE-CLI-03, SKE-CLI-04
**Success Criteria** (what must be TRUE):
  1. Operator can run benchmark evaluation with `--static-evidence auto`, while `--static-evidence none` remains the default.
  2. Operator can find the static evidence JSON sidecar and evidence artifact directory beside the configured trace output path.
  3. Operator can run benchmark evaluation normally when static evidence is skipped, unavailable, partial, or failed; static evidence does not change benchmark exit-code semantics except for invalid CLI usage.
  4. Researcher can read a human-facing static evidence report or summary explaining aggregate status, artifact manifest, routing decisions, extracted metadata/ISA presence, unsupported states, and claim boundaries.
**Plans**: TBD

### Phase 77: Documentation, Guardrails, And Live Validation
**Goal**: Researchers and maintainers can interpret, test, and bound static evidence claims with documented CPU-safe coverage and at least one bounded live RDNA 4 validation record when the environment supports it.
**Depends on**: Phase 76
**Requirements**: SKE-DOCS-01, SKE-DOCS-02, SKE-DOCS-03, SKE-DOCS-04, SKE-DOCS-05
**Success Criteria** (what must be TRUE):
  1. Researcher can read documentation explaining how to enable static evidence, interpret collected, partial, unavailable, unsupported, failed, and skipped states, and archive static sidecars.
  2. Researcher can read claim-boundary documentation stating static evidence is diagnostic static-analysis evidence, not correctness, performance, timing, score, paper-parity, or leaderboard authority.
  3. Maintainer can run CPU-safe tests with fixture artifacts, fake routed tools, and fake extractor outputs for schema, discovery, parsing, CLI sidecar writing, and claim guardrails.
  4. Maintainer can inspect a bounded live ROCm validation artifact for at least one RDNA 4 HIP/C++ build when required ROCm build tools and device/runtime access are available.
  5. Researcher can see CDNA 3, CDNA 4, Triton, RGA-rich resource parsing, and paper-scale static coverage marked unsupported, partial, or deferred unless direct evidence exists.
**Plans**: TBD
