# Phase 136 Plan: RDNA4 Validation Scope and Long-Tail Exclusion Controls

## Goal

Define the RDNA4 `gfx1200` benchmark-grade validation denominator and add
default-off, explicit long-tail exclusion configuration to the dataset runner
without weakening denominator, readiness, execution-closure, or claim
boundaries.

## Tasks

1. Define RDNA4 validation scope documentation.
   - Add a phase-local validation scope artifact describing selected dataset
     roots, expected denominator, output artifact roots, run commands, and claim
     boundaries.
   - Record that Phase 136 does not execute the full dataset and does not
     upgrade public claims.
   - Identify the prior validation records that may justify temporary
     long-tail exclusions, using exact repository artifact names.

2. Add an importable long-tail exclusion model.
   - Add a small `src/sol_execbench/core/dataset/` helper for parsing and
     validating exclusion configuration.
   - Support problem-level, workload-level, and shard-level exclusions with
     reason, evidence reference, owner/status metadata, and optional expiry.
   - Reject ambiguous or malformed entries with clear errors.
   - Preserve default behavior when no config is supplied.

3. Wire exclusion configuration into `scripts/run_dataset.py`.
   - Add an explicit CLI flag for the exclusion config.
   - Apply exclusions before execution selection while preserving denominator
     records for excluded entries.
   - Include exclusion config refs/checksums in run provenance and evidence
     refs.
   - Keep trace reuse and derived phases deterministic when the exclusion config
     changes.

4. Preserve closure/readiness visibility.
   - Extend closure or selection records so excluded entries are visible as a
     non-pass status such as `excluded_long_tail`.
   - Ensure excluded entries do not count as passed, failed, missing, or
     silently omitted.
   - Include exclusion summaries in readiness/denominator/closure outputs where
     those reports already expose selected or omitted workload accounting.

5. Add tests and documentation guardrails.
   - Add CPU-safe tests for config parsing, CLI wiring, default-off behavior,
     checksum/provenance sensitivity, and closure visibility.
   - Cover problem-level, workload-level, and shard-level exclusions.
   - Add guardrails proving exclusions do not imply RDNA4 validation success,
     paper parity, leaderboard authority, CDNA3/MI300X validation, or CDNA4
     validation.

## Risks

- Execution closure models may reject new fields because several report
  schemas are intentionally strict; additions must be explicit and tested.
- Existing run reuse logic may need exclusion-config checksums to avoid reusing
  traces from a different denominator.
- The user mentioned CDNA4 real-validation records, while current public
  project docs mark CDNA4 as deferred. Verify exact artifact names before
  committing any evidence-reference wording.
- Exclusions must remain temporary accounting controls, not a way to mark
  hard cases as validation passes.
