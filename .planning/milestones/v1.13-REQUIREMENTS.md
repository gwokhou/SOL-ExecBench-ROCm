# Requirements: SOL ExecBench ROCm Port v1.13

**Defined:** 2026-05-25
**Milestone:** v1.13 ROCm Runtime Evidence and Environment Diagnostics
**Status:** PLANNING
**Source:** `.planning/notes/2026-05-23-rocm-systems-gpuopen-enhancements.md`

## Goal

Make benchmark runs reproducible and diagnosable by attaching ROCm/GPU
environment evidence and fast preflight diagnostics without changing SOL
ExecBench correctness, timing, scoring, or v1.12 consumer defaults.

## Compatibility Position

v1.13 is a backward-compatible evidence milestone.

- Keep `sol_execbench.evaluator_contract.v1` and contract version `1.0`.
- Preserve primary `sol-execbench <problem_dir> --solution <solution-path>`
  behavior and existing trace semantics.
- Add only optional evidence metadata and optional contract capabilities.
- Do not require `amd-smi`, `rocminfo`, `rocm_agent_enumerator`, or ROCm GPU
  tools for ordinary non-profiled benchmark execution.
- Do not make environment evidence part of correctness, SOL/SOLAR scoring, or
  pass/fail status.

## Requirements

### Environment Evidence

- [x] **ENV-01**: Add an internal environment snapshot model that can represent
  ROCm version, HIP runtime visibility, PyTorch ROCm metadata, GPU identity,
  `gfx` target, visible devices, and collection provenance.
- [x] **ENV-02**: Collect environment evidence from available local sources,
  including `amd-smi`, `rocminfo`, `rocm_agent_enumerator`, and PyTorch ROCm
  metadata, with bounded timeouts and structured unavailable/error statuses.
- [x] **ENV-03**: Persist environment evidence as optional run metadata without
  changing canonical correctness, timing, scoring, or public problem schemas.
- [x] **ENV-04**: Expose a stable optional contract capability such as
  `runtime.evidence.v1` so consumers can detect v1.13 evidence support while
  v1.12-compatible contract validation remains possible.

### Diagnostics CLI

- [ ] **DIAG-01**: Add a GPU-safe diagnostic command, `sol-execbench doctor` or
  `sol-execbench env-snapshot`, that emits machine-readable environment and
  tool readiness evidence.
- [ ] **DIAG-02**: The diagnostic command distinguishes missing tools, missing
  GPU visibility, ROCm runtime failures, and successful checks using stable
  statuses and remediation text.
- [ ] **DIAG-03**: The diagnostic command must be runnable independently of a
  benchmark problem directory and candidate solution.

### Preflight Smoke Checks

- [ ] **SMOKE-01**: Add lightweight ROCm/HIP preflight checks for tool
  availability, HIP runtime initialization, device visibility, simple device
  memory behavior where feasible, and event timing readiness.
- [ ] **SMOKE-02**: Preflight checks are opt-in for normal users and
  auto-skipped in ordinary CPU-only test environments.
- [ ] **SMOKE-03**: Add architecture-aware test marker support for RDNA 4 and
  CDNA 3 checks without forcing unavailable hardware in default pytest runs.

### Consumer Compatibility

- [x] **COMPAT-01**: Existing v1.12-style contract consumers, including
  hip-playground-style readiness probes, can continue to validate the contract
  unless they explicitly require v1.13 evidence capability.
- [x] **COMPAT-02**: Old result fixtures and trace JSONL fixtures continue to
  parse without environment evidence.
- [x] **COMPAT-03**: Documentation explains which fields are optional evidence,
  which commands are diagnostic-only, and why evidence metadata does not affect
  correctness or scoring.

## Out of Scope

| Feature | Reason |
|---------|--------|
| `rocprofv3` profile artifact collection | Planned for v1.14 after the environment evidence surface is stable. |
| RGA/code-object/static ISA analysis | Planned for a later static kernel evidence milestone. |
| Contract version bump to `1.1` | Not needed for backward-compatible v1.13 evidence. |
| Making confirmed benchmark claims require environment evidence | May be a consumer policy decision, but v1.13 does not force it. |
| Full CDNA 3 / MI300X validation claim | v1.13 improves diagnostics but does not complete full hardware validation. |
