# 191-01 Plan: Structured profile metrics and diagnostic bottleneck hints

## Objective

Expand `profile_summary.sidecar.v1` with structured workload/kernel profiling
evidence and conservative AMD-oriented bottleneck hints derived from bounded
CSV/JSON profiler artifacts, while preserving diagnostic-only authority
boundaries and HIP-facing fixtures.

## Source Items Covered

- PSUM-01: profile summary contains workload and kernel-level metric records
  derived from bounded profiler artifacts, including duration, dispatch/count,
  and artifact coverage fields.
- PSUM-02: profile summary emits stable bottleneck hints for compute,
  memory/L2, LDS, launch/dispatch, insufficient-counter, and unknown states.
- PSUM-03: profile summary remains diagnostic-only and cannot claim benchmark
  authority.
- D-01: parse CSV and JSON artifacts only.
- D-02: cite `.rocpd` and database artifacts without parsing.
- D-03: use canonical trace/profile metadata where bounded and available.
- D-04: use conservative closed bottleneck taxonomy.
- D-05: do not emit fine-grained occupancy/VGPR/SGPR/cache/bandwidth claims.
- D-06: degrade missing or insufficient counters to
  `insufficient_counters`/`unknown`.
- D-07: update schema, fixtures, docs, and tests.
- D-08: keep new fields diagnostic-only; they cannot become correctness,
  timing, performance, score, evidence-tier, release-gate, cutover,
  paper-parity, leaderboard, or claim-upgrade authority.
- D-09: document HIP-facing bounded adapter input semantics and downgrade
  behavior for unknown hint categories.
