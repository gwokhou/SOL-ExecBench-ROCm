# Requirements: SOL ExecBench ROCm Port v1.30

**Defined:** 2026-06-07
**Core Value:** Evaluate LLM-generated GPU kernels correctly and reproducibly on
AMD ROCm hardware while preserving the benchmark semantics and rigor of SOL
ExecBench.

## v1.30 Requirements

### Validation Scope And Orchestration

- [x] **RDNA4-VAL-01**: RDNA4 `gfx1200` validation scope defines the exact
  dataset denominator, migrated dataset inputs, artifact output roots, and
  claim boundaries before long-running jobs start.
- [x] **RDNA4-VAL-02**: Validation orchestration supports many-hour runs by
  preserving bounded logs, recording progress checkpoints, allowing resume from
  existing traces/artifacts, and polling healthy background jobs instead of
  killing them solely due to duration.
- [x] **RDNA4-VAL-03**: Full adapted pytest validation runs on RDNA4 or records
  every skip/failure as an accepted hardware/tooling boundary.
- [x] **RDNA4-VAL-04**: Full dataset validation runs on RDNA4 with complete
  problem/workload denominator accounting and no silent omissions.

### Long-Tail Exclusion Accounting

- [x] **RDNA4-EXCL-01**: `scripts/run_dataset.py` accepts an explicit
  configuration for temporarily excluding known super-long-tail shards,
  problems, or workloads from execution.
- [x] **RDNA4-EXCL-02**: Exclusion configuration cites the prior real-validation
  record or local evidence that justifies each excluded shard/problem/workload,
  including any CDNA3/CDNA4 record names exactly as verified in repository
  artifacts.
- [x] **RDNA4-EXCL-03**: Excluded entries remain visible in readiness,
  denominator, execution-closure, and evidence-bundle reports as excluded or
  deferred, not passed and not missing.
- [x] **RDNA4-EXCL-04**: Exclusion configuration is reversible and does not
  change default dataset semantics unless explicitly supplied by the caller.

### Timing And Environment Evidence

- [x] **RDNA4-TIME-01**: RDNA4 validation records environment evidence for
  ROCm, PyTorch, HIP, GPU architecture, device identity, driver/tool versions,
  and tool availability.
- [x] **RDNA4-TIME-02**: Locked-clock status is captured before benchmark-grade
  timing; unlocked-clock evidence is classified as a blocker or accepted
  non-authoritative timing boundary.
- [x] **RDNA4-TIME-03**: `rocprofv3` timing evidence is collected or every
  missing profiler artifact is classified with a concrete blocker.
- [x] **RDNA4-TIME-04**: Timing stability reports classify dispersion,
  backend, profiling-overhead risk, and repeatability for RDNA4 traces.

### Derived Reports And Evidence Bundle

- [x] **RDNA4-REP-01**: AMD-native score reports are generated from RDNA4 trace
  evidence and classify scored, unscored, degraded, failed, and excluded cases.
- [x] **RDNA4-REP-02**: AMD SOL/SOLAR bound artifacts are generated from RDNA4
  trace evidence with hardware-model refs, coverage semantics, and warnings.
- [x] **RDNA4-REP-03**: Consistency, stability, claim-upgrade, and trust-summary
  reports are generated for the RDNA4 evidence set.
- [x] **RDNA4-REP-04**: A reproducible RDNA4 evidence bundle references traces,
  timing artifacts, score reports, bound artifacts, static evidence, exclusion
  configs, logs, environment sidecars, and checksums.

### ROCm Category Coverage

- [x] **RDNA4-CAT-01**: HIP/C++ and Triton ROCm category coverage runs on RDNA4
  or records accepted dependency/hardware boundaries.
- [x] **RDNA4-CAT-02**: hipBLAS, MIOpen, CK, and rocWMMA category coverage runs
  on RDNA4 or records accepted dependency/hardware boundaries.
- [x] **RDNA4-CAT-03**: Category-level failures are classified as correctness,
  compile, runtime, dependency, timeout, expected skip, or accepted known gap.
- [x] **RDNA4-CAT-04**: Public docs and tests prevent category readiness from
  being reported as benchmark-grade validation unless required evidence exists.

### Claim Closure

- [x] **RDNA4-CLAIM-01**: `docs/CLAIMS.md`, README validation status, and
  release/readiness docs are updated to state the RDNA4 benchmark-grade result
  only within the archived evidence boundary.
- [x] **RDNA4-CLAIM-02**: Guardrail tests prevent RDNA4 evidence from implying
  NVIDIA B200 equivalence, upstream SOLAR equivalence, hosted leaderboard
  authority, CDNA3/MI300X validation, or CDNA4 validation.
- [x] **RDNA4-CLAIM-03**: Any remaining RDNA4 failures, exclusions, or deferred
  items are documented as blockers or accepted known gaps before the milestone
  is considered complete.

## vNext Requirements

### Hardware Validation

- **CDNA3-VAL-01**: Complete benchmark-grade MI300X validation with exact
  hardware evidence, locked-clock timing, score reports, and accepted skip
  boundaries.
- **CDNA4-VAL-01**: Validate NVFP4/MXFP4 or CDNA4-specific compatibility paths
  on suitable CDNA4-class hardware when available.

## Out of Scope

| Feature | Reason |
|---------|--------|
| NVIDIA B200 equivalence | RDNA4 validation does not validate NVIDIA hardware or original SOL-Score roofline authority. |
| Upstream SOLAR equivalence | AMD SOL/SOLAR artifacts remain project-scoped unless separately validated against upstream SOLAR semantics. |
| Hosted leaderboard authority | Operational leaderboard support is outside this validation milestone. |
| CDNA3-family, CDNA4, or native-host ROCm validation expansion | RDNA4 closure must not expand unrelated hardware or host-validation claims. |
| CDNA3/MI300X validation upgrade | CDNA3 requires exact MI300X evidence and separate acceptance criteria. MI300X and MI308X are sibling GPU products, and MI308X evidence must not be reported as completed MI300X validation. |
| Actual MI300X full-suite execution under CDNA3 in v1.28 | v1.28 was readiness and documentation work only; v1.30 remains RDNA4-scoped. |
| CDNA4 validation | Suitable CDNA4-class hardware is not available in this milestone. |
| Dataset redistribution | NVIDIA/SOL-ExecBench original or derivative dataset content must remain local-only. |
